# Задание 1. Анализ и планирование

<aside>

"Теплый дом" - это комплексное решение для управления "умными" устройствами, спроектированая на основе микросервисной архитектуре. 
Система предоставляет пользователям системы широкие возможности по управлению устройствами и получения телеметрии от подключенных устройсв, а также добавлению устройство предоставляемых партнерами. Система предоставляет администраторам широкие возможности по управлению и определению внештатный систуаций для последующего их устранения. 

</aside>

### 1. Описание функциональности монолитного приложения

**Управление отоплением:**

- Пользователи могут удалённо включать/выключать отопление в своих домах через датчики
- Пользователи могут проверить температуру по датчикам
- Система поддерживает обращение по HTTP с помощью JSON запросов

**Мониторинг температуры:**

- Пользователи могут просматривать текущую температуру в своих домах через веб-интерфейс.
- Система получает данные о температуре с датчиков, установленных в домах


### 2. Анализ архитектуры монолитного приложения

* Язык программирования: Go
* База данных: PostgreSQL
* Архитектура: Монолитная, все компоненты системы (обработка запросов, бизнес-логика, работа с данными) находятся в рамках одного приложения.
* Взаимодействие: Синхронное, запросы обрабатываются последовательно.
* Масштабируемость: Ограничена, так как монолит сложно масштабировать по частям.
* Развертывание: Требует остановки всего приложения.

### 3. Определение доменов и границы контекстов

* Домен: "Управление устройствами"
	- поддомен управления сенсорами
		+ контекст запись данных с сенсоров
* Домен: "Мониторинг температуры"
	- поддомен отслеживание данных о температуре

### **4. Проблемы монолитного решения**

- Сложность масштабирования, необходимость обновлять все приложение, сильная связанность
- Монолитная архитектура, при необходимости добавления функциональности придется дорабатывать в монолите, что приведет к необходимости проверять все и со всем, даже при незначительных улучшениях
- Синхронное взаимодействие, нет возможности добавить асинхронные способы получения данных, например, при необходимости непрерывного получения значений
- Нет возможности параллельной разработки, для увеличения эффективности работы команд разработки и снижению рисков
- При изменениях высокий риск ошибок. Исправления одного может поломать, что угодно
- Сложность развертывания, обновление системы требует приезда специалиста
- Трудность управления командой, изменения одной команды может пересекаться с тем, над чем работает другая
- При обновлении требуется остановка всего приложения
- Ограниченность выбора технологической среды, не всегда есть возможность использовать сторонние решения обеспечивающие более высокую эффективность
- Отказ одной части зачастую приводит к отказы всей системы
- Тесная связанность компонентов приводит с снижению производительности

# Задание 3. Разработка ER-диаграммы

- Идентификация сущностей
+ Пользователь (User)
+ Расположение (Location)
+ Устройство (Device)
+ Тип устройства (DeviceType)
+ Телеметрия (TelemetryData)
+ Команда управления (Command)
+ Тип команды управления (CommandType)

- Определение атрибутов: 

+ Сущность «Устройство»:
* id — уникальный ID устройства.
* type_id — ID типа устройства — внешний ключ к таблице DeviceType.
* location_id — ID расположения, внешний ключ к таблице Location.
* serial_number — серийный номер устройства.
* status — текущее состояние устройства (включено/выключено).
* created_at — время подключения.
* name — название устройства.

+ Сущность «Пользователь»:
* id — уникальный идентификатор пользователя.
* login — логин пользователя.
* password — пароль пользователя.
* fio — ФИО польователя.
* phone — контактный телефон.
* email — почта.
* push_token — токен для отправки PUSH-уведомлений.

+ Сущность «Расположение»:
* id — уникальный ID расположения.
* user_id — ID пользователя — внешний ключ к таблице User.
* location_type — тип расположения.
* name — название расположения.
* room — дополнительное название для комнаты.

+ Сущность «Тип устройства»:
* id — уникальный ID устройства.
* device_id — ID устройства — внешний ключ к таблице Device.
* type — тип устройства.
* name — название типа.

+ Сущность «Телеметрия»:
* id — уникальный ID телеметрии.
* device_id — ID устройства — внешний ключ к таблице Device.
* data — информация по типу телеметрии.
* date_time — время замера.
* name — название телеметрии.

+ Сущность «Команда управления»:
* id — уникальный ID команды управления.
* device_id — ID устройства, внешний ключ к таблице Device.
* type_id — ID типа команды, внешний ключ к таблице CommandType.
* data — информация по команде.

+ Сущность «Тип команды управления»:
* id — уникальный ID типа команды управления.
* command_id — ID команды — внешний ключ к таблице Command.
* type — тип команды (дополнительное описание типа команды).
* name — название типа команды.


Описание связей: Определите связи между сущностями (один-к-одному, один-ко-многим, многие-ко-многим).
- Пользователь — Расположение: один пользователь может иметь доступ к нескольким расположениям, но каждый расположение связано только с одним пользователем.
- Расположение — Устройство: в одном расположении может быть несколько устройств, и каждое устройство принадлежит только одному расположению.
- Устройство — Тип Устройства: у одного устройства может быть один тип устройства.
- Устройство — Телеметрия: одно устройство может генерировать множество записей телеметрии.
- Устройство — Команда: в одном устройстве может быть несколько команд управления.
- Команда — Тип Команда: одна команда может иметь только один тип команды.

### 2. Документация API

REST API. Он хорошо подходит для синхронного взаимодействия, когда клиенту (в данном случае — другому микросервису) нужен немедленный ответ на запрос. Большенство запросов анализирует Response который получают по запросу.

AsyncAPI. Его используют для асинхронного взаимодействия, когда клиенту не нужно ждать ответа немедленно (например, отправка уведомления о новом измерении температуры).
При необходимости продумывается решение в котором, можно будет настраивать расписание получения показаний, например, как в умных часах, в период бега каждую секунду отправлять на почту пороговые значения.

*  Взаимодействие
  - Внешние системы подключаются через REST или WebSocket:
   - Stream Devices - взаимодействие по WebSocket (потоковые данные, например, камера или сложные данные).
   - Single Devices - взаимодействие по REST (разовые показания, температура, сенсоры движения).
   - Control System - управление устройствами через REST (команды управления).
   - Системы уведомлений (SMS, Push, Email) - через REST/SMTP/SMPP/IMAP.

```
	// If no location is provided, use a default based on sensor ID
	if location == "" {

		}
	}
```
